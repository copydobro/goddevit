# FnKey — GTM Audit Plan

> Цель: провести полный GTM-аудит на реальном проекте, получить первый кейс.
> Данные: только открытые источники (GitHub, PH, dev.to, личный блог).
> Вопросы 5-6 из брифа закрываем гипотезами на основе паттернов.

---

| Источник | Что берём |
|----------|-----------|

## Этап 1 — Позиционирование и ICP
`Файл: 02_POSITIONING_ANALYSIS.md`

Вопросы:
- Кто реальный ICP: все macOS-юзеры или узкая ниша devs + Claude Code?
- Как FnKey себя описывает vs как его описывают другие (PH комменты, awesome-mac)?
- Есть ли отличие от конкурентов на уровне слов или только на уровне tech?
- Словарь: использует ли README язык пользователя или язык разработчика?

Источники: README, PH listing + комменты, dev.to пост "Voice Transcription is a Superpower"

---

## Этап 2 — Упаковка (README + точки входа)
`Файл: 03_PACKAGING_AUDIT.md`

Вопросы:
- README: Time to First Value — сколько шагов до "оно работает"?
- Есть ли GIF/видео демо?
- Есть ли 1-liner который можно скопировать в твит?
- Механика захвата: что происходит после того как человек нашёл продукт — куда идти, на что подписаться?
- Visit Website на PH ведёт в README — сколько людей отвалилось на этом шаге?

Источники: GitHub README, PH listing (Visit Website link), конкурентный анализ

## Этап 3 — История запусков
`Файл: 04_LAUNCH_AUDIT.md`

Вопросы:
- PH: что сработало (107 upvotes при 35 фолловерах X) — откуда пришли голоса?
- dev.to: какой охват дали 2 поста реально?
- Что не было сделано: Reddit (r/macapps, r/rust), HN, Discord-сообщества
- Storytelling: есть ли внятная история "почему я это построил" упакованная для разных каналов?

Источники: PH listing, dev.to, awesome-mac, данные по соцсетям

---

## Этап 4 — Текущий GTM-статус
`Файл: 05_GTM_GAPS.md`

Финальный gap analysis: что сломано, что отсутствует, приоритет по импакту.
Формат: таблица с колонками — Фаза / Проблема / Импакт (H/M/L) / Быстрое решение.

Минимум 5 gaps (по нашей гарантии).

---

## Критерий готовности

Аудит считается завершённым когда:
- [ ] Найдено ≥5 конкретных, измеримых gaps
- [ ] Каждый gap имеет конкретное действие (не "улучшить README")
- [ ] 30-дневный план реалистичен для одного фаундера без команды
- [ ] Отчёт можно показать следующему клиенту как пример работы
