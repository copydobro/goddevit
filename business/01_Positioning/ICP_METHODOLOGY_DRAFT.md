# ICP Methodology — Рабочий черновик
> Статус: в разработке
> Контекст: выявлено в процессе GTM-аудита FnKey (2026-03-25)
> Цель: стать основой для нового скила `icp-definition`

---

## Почему мы это пишем

Во время аудита FnKey мы написали:

> "Реальный ICP: разработчики использующие Claude Code или Cursor, которые думают быстрее чем печатают."

И обосновали это одним dev.to постом автора.

Это слабо. И это создало внутреннее противоречие в отчёте:
- ICP = технические разработчики
- Рекомендации = онбординг wizard "для нетехнической аудитории", лендинг "нетехническая аудитория уходит"

Если ICP — технари, то 9 шагов с `brew` и `.zshrc` для них не барьер.
Если ICP — широкая аудитория, то Homebrew как основной канал дистрибуции противоречит этому.

**Вывод:** у нас нет методологии для определения ICP. Мы угадывали.

---

## Фундаментальная проблема

Существующие скилы не закрывают эту задачу:

- `positioning-messaging` — начинается ПОСЛЕ того как ICP известен
- `problem-definition` — про проблему, не про человека
- `gtm-customer-language-ru.md` — отличное исследование языка для ЗАРАНЕЕ ИЗВЕСТНОГО ICP (tech founders с dev tools). Отвечает на "как они говорят" — не на "как мы их нашли"

Нам нужен инструмент для шага ДО всего этого.

## Две принципиально разные ситуации

### Ситуация A — Есть хоть какая-то аудитория (post-launch)

Примеры сигналов:
- GitHub stargazers — кто они, какие у них репозитории, на кого подписаны
- Product Hunt комментарии — язык, описание своей ситуации, use cases
- GitHub issues — что просят, как формулируют проблему
- Упоминания в соцсетях — кто делится и почему

Метод: смотришь паттерны реальных людей → выводишь ICP из данных

Для FnKey это возможно:
- Stargazers: github.com/evoleinik/fnkey/stargazers
- PH комментарии: 107 апвоутов = ~107 человек с мнением
- Awesome-mac листинг: какая категория, кто рядом

### Ситуация B — Аудитории нет вообще (pre-launch)

Нет данных. Нет паттернов. Только продукт и гипотеза фаундера.

Три источника для ICP-гипотезы:

**Источник 1: Фаундер = первый ICP**
Большинство технических продуктов строится под собственную боль.
Вопрос: кто ещё находится в точно такой же ситуации что и я в момент когда мне это понадобилось?
Для FnKey: Евгений — CTO, использует Claude Code ежедневно, печатает сложные промпты. Первая гипотеза = люди как он.

**Источник 2: Ищешь проблему, не продукт**
Не "кто хочет голосовой ввод" — а "кто прямо сейчас жалуется на конкретный struggling moment".
Идёшь туда где проблема обсуждается: Reddit, HN, Discord, Slack-комьюнити.
Ищешь реальные цитаты. Смотришь кто пишет. Это и есть ICP.

Пример: `gtm-customer-language-ru.md` — именно такое исследование.
50+ цитат с платформ → паттерн → ICP сформулирован до первого пользователя.

**Источник 3: Конкурентная аудитория как прокси**
Кто платит за Wispr Flow ($240/год)? Кто пишет в их PH-комментариях?
Кто оставляет отзывы на superwhisper в App Store?
Это рынок. Их язык, их боли, их профили — прокси для ICP до того как есть единственный собственный пользователь.

## Что должен делать ICP-скил

### Два режима работы

```
MODE A: Post-launch signal mining
─────────────────────────────────
Вход:  GitHub stargazers + PH comments + issues + упоминания
Шаги:
  1. Собрать профили (кто они профессионально)
  2. Найти общие паттерны (роль, тип продукта, стек)
  3. Выделить язык которым описывают use case
  4. Сформулировать ICP-гипотезу
  5. Проверить: совпадают ли каналы дистрибуции с ICP?
Выход: ICP-документ с доказательствами из данных


MODE B: Pre-launch hypothesis
──────────────────────────────
Вход:  описание продукта + профиль фаундера + конкуренты
Шаги:
  1. Фаундер как первый ICP — кто ещё в его ситуации?
  2. Поиск struggling moment на форумах/Reddit/HN
  3. Анализ конкурентной аудитории (PH, App Store, Twitter)
  4. Сформулировать 2-3 ICP-гипотезы
  5. Ранжировать по доступности канала
Выход: ICP-гипотеза + список где её проверить
```

### Обязательная проверка на противоречия

После определения ICP скил должен флагировать несоответствия:

```
ПРОВЕРКА: ICP vs Рекомендации
─────────────────────────────
Если ICP = технари:
  → Homebrew — ок
  → 9 шагов установки — допустимо
  → Лендинг нужен но не для "нетехнических уходят"
  → Onboarding wizard — nice-to-have, не критично

Если ICP = широкая аудитория:
  → Homebrew — барьер, нужен .dmg / App Store
  → 9 шагов — критично, нужен wizard
  → Лендинг — обязателен, первая точка входа
  → Onboarding wizard — критично
```

## Шаблон ICP-документа (выход скила)

```markdown
# ICP: [название сегмента]

## Кто они
[Роль, тип компании/проекта, технический уровень]

## Ситуация (когда ищут решение)
[Конкретный момент. Не "хотят голосовой ввод" а
"используют Claude Code ежедневно, замечают что
печатают урезанные версии своих мыслей"]

## Что делали ДО
[Какую альтернативу использовали или как справлялись]

## Язык которым описывают проблему
[Дословные цитаты или близкие к ним формулировки]

## Где их найти
[Конкретные каналы: какие subreddits, HN теги,
GitHub topics, PH категории]

## Кто точно НЕ ICP
[Явное исключение — помогает не размывать фокус]

## Источник данных
[Откуда мы это знаем: stargazers / PH / форумы /
гипотеза фаундера / конкурентная аудитория]

## Уровень уверенности
[Высокий (данные) / Средний (прокси) / Гипотеза]
```

## Открытые вопросы для исследования

1. **Как быстро прочитать 100+ stargazers профилей?** Вручную — долго. Нужен паттерн что смотреть в первую очередь (bio, pinned repos, following).

2. **Что делать если stargazers разнородны?** Несколько сегментов → нужно выбрать фокус. Критерий выбора?

3. **Как валидировать ICP-гипотезу без интервью?** Прокси-метрики: конверсия по каналу, retention, кто возвращается.

4. **Множественный ICP — когда это ок?** Wispr Flow бьёт в разных людей. FnKey пока не может себе это позволить — слишком мало ресурсов.

5. **Когда ICP меняется?** По мере роста продукт может найти неожиданный сегмент. Как это отследить из данных?
