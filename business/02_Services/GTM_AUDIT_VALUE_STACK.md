# GTM Audit — Полный список ценностей для клиента
> Основан на реальном кейсе: FnKey (2026-03-25)
> Аудитория: мы сами — чтобы знать что продаём и уметь это объяснить

---

## Контекст: кто наш клиент

Dev или OSS-фаундер который:
- Построил продукт, запустил — "тишина"
- Не знает где конкретно сломано: продукт, упаковка или дистрибуция
- Не хочет нанимать агентство на 3 месяца, платить $5K+ за "стратегию"
- Хочет конкретный список: вот что не так, вот что делать, в каком порядке

---

## Ценности — полный список

### I. ДИАГНОСТИКА: "Теперь я знаю что сломано"

**1. Знаешь кто твой реальный ICP — с доказательствами, не с догадками**
Большинство фаундеров описывают свой ICP как "разработчики" или "все кто хочет X".
Мы выводим ICP из данных: stargazers, PH-комментарии, GitHub issues, цитаты с форумов.
Результат: конкретный профиль с уровнем уверенности (Высокий / Средний / Гипотеза)
и точным языком которым эти люди описывают свою проблему — дословно, не в пересказе.

*Пример FnKey: "Solo developer, Claude Code daily, 20-70 prompts/day, уже пробовал Wispr Flow, недоволен багами" — HIGH confidence, 35+ цитат с 6 платформ.*

**2. Знаешь второй слой боли — тот что ты не видел**
Мы ищем не только очевидную боль из твоих описаний, но и боль которую пользователи
описывают между строк. Часто это важнее первого слоя.

*Пример FnKey: первый слой = "печатать медленно". Второй слой = "фильтрую мысли пока печатаю — Claude получает урезанный контекст". Второй слой — это и есть настоящий продающий аргумент.*

**3. Знаешь где конкретно отваливаются пользователи**
Не "конверсия низкая" — а "вот конкретная точка: PH → README → первый экран с terminal commands.
Здесь FRR падает. BRR высокий потому что конкуренты устанавливаются в 2 клика."

**4. Знаешь что делают конкуренты которых ты не замечал**
Включая новых entrants которые появились после твоего последнего анализа.

*Пример FnKey: Claude Code /voice вышел в марте 2026 — новый конкурент которого не было в изначальном конкурентном анализе.*

**5. Знаешь откуда пришли твои лучшие пользователи**
И почему — что в твоём продукте или канале сработало, даже если ты этого не планировал.

*Пример FnKey: 107 upvotes при 35 фолловерах = Antler-сеть, использованная неформально один раз. Это было неосознанным borrowed каналом.*

---

### II. УПАКОВКА: "Теперь у меня есть готовый текст"

**6. Готовый README hero — не "улучши текст", а конкретные строки**
Не "добавь value prop". А: вот точные строки, откуда они взяты, почему они работают.
Можно скопировать и вставить.

*Пример FnKey: "AI coding tools removed the bottleneck of writing code. They created a new one: how fast you can type prompts. Hold Fn. Speak everything in your head. Claude Code gets the full context — not the filtered version."*

**7. Готовый PH tagline**
Текущий vs предлагаемый. Одно предложение. Объяснение почему старый не работал.

*Пример FnKey: "with Deepgram stream" → "voice input for Claude Code and Cursor, zero delay, free"*

**8. Готовый competitive block для README**
Три аргумента против главных конкурентов — цена, надёжность, скорость.
Написан языком который ICP уже использует, не маркетинговым.

*Пример FnKey: "Free forever. Wispr Flow costs $240/year and cuts out mid-sentence. Claude Code /voice has 500ms lag. FnKey: streaming, open source, mic active only while you hold the key."*

**9. Реструктурированный Install / онбординг**
Пошагово: было N шагов → стало M шагов. Объяснение каждого убранного шага.

*Пример FnKey: 9 шагов → 6. Groq как quick start (60 сек, без карты) вместо Deepgram.*

**10. Готовый копирайт лендинга — 5 экранов**
Hero, Pain, Speed, Social Proof, Competitive — с принципами и объяснением каждого решения.
Не wireframe — готовый текст для разработки.

---

### III. ДИСТРИБУЦИЯ: "Теперь я знаю куда идти"

**11. Список каналов которые ты не использовал — с приоритетами**
Не "попробуй Reddit". А: r/ClaudeAI, r/cursor, r/LocalLLaMA — с конкретным углом для каждого.
С объяснением почему именно эти, а не другие.

**12. Готовый Show HN пост**
Заголовок, первый абзац, технические детали, время публикации.
Данные по историческому импакту для аналогичных продуктов.

**13. Шаблон для каждого канала**
Reddit, HN, LinkedIn, Antler/нетворк — отдельный текст под каждый.
Не одно сообщение для всех.

**14. Стратегия следующего PH-запуска**
Coming Soon страница, план pre-warm, что публиковать и когда в launch day.
Включая post-launch контент — то что большинство фаундеров пропускают.

---

### IV. СОЦИАЛЬНОЕ ДОКАЗАТЕЛЬСТВО: "Теперь у меня есть proof"

**15. Список реальных цитат от пользователей — дословно**
Из forums, Reddit, GitHub issues, PH-комментариев — язык которым реальные люди описывают
боль которую решает твой продукт. Готов для использования в landing page и в постах.

*Пример FnKey: 35+ цитат с 6 платформ. Включая Karpathy, OpenAI exec, директора ML Wikimedia.*

**16. Публичные high-profile endorsers которых ты не замечал**
Люди которые публично используют или описывают твою проблему — не требуют разрешения,
публичные высказывания, можно цитировать как market validation.

**17. Шаблон для сбора testimonials**
Кому писать (конкретные имена из PH-комментариев), что писать, как формулировать запрос.

---

### V. ПЛАН: "Теперь я знаю что делать в понедельник утром"

**18. 30-дневный план с конкретными днями**
Не "в первую неделю улучшите README".
А: "День 1 — вот три строки заменить, вот куда вставить. 2 часа."

**19. Матрица быстрых побед**
Quick wins (высокий импакт, низкое усилие) отделены от стратегических.
Первые 5 действий которые можно сделать сегодня.

**20. Временные затраты**
Сколько часов в неделю нужно. Без агентства. Без бюджета. Без команды.

*Пример FnKey: 18 часов за 30 дней. ~35 минут в день.*

**21. Критерии успеха через 30 дней**
Конкретные метрики: базовое значение → цель. Не "вырасти" — а "43 → 150 звёзд".

---

### VI. МЕТА-ЦЕННОСТИ: то что не написано в deliverables

**22. Экономия 30-50 часов исследования**
Мы прошли весь путь: от сырых данных до готового текста. Solo-фаундер потратил бы на это
4-6 недель неравномерной работы между релизами. Мы делаем это за 7 рабочих дней.

**23. Внешний взгляд без конфликта интересов**
Фаундер не может объективно оценить свой продукт. Мы видим его глазами нового пользователя.
Это не консультация — это диагностика от людей которые не влюблены в продукт.

**24. Методология, а не мнение**
Каждый вывод имеет источник: откуда данные, как мы к этому пришли, уровень уверенности.
Ты знаешь не только что делать — но и почему именно это, и насколько мы уверены.

**25. Документация которая остаётся**
Все артефакты — в папке которую можно передать следующему контрактору, коинвестору,
или использовать для следующего GTM-цикла через 6 месяцев.

### VII. ДОПОЛНИТЕЛЬНЫЕ ЦЕННОСТИ: выявлены через Value Architecture 3.0

**26. Стоимость бездействия по каждому gap**
Каждый незакрытый критический gap имеет измеримую цену.
Мы не просто говорим "тебе нужно выйти на HN" — мы показываем что именно ты теряешь каждый месяц пока не выходишь.
Это язык инженера: не "это важно", а "вот конкретные единицы потерь".

*Пример FnKey: Show HN не сделан → ~150 stars упущено → при текущем органическом темпе это 4-6 месяцев роста которые уже не вернуть.*

> ⚠️ Идея в разработке: Cost of Inaction Matrix — отдельный артефакт внутри аудита.
> Для каждого gap: если не сделать → теряешь X звёзд / Y пользователей / Z месяцев роста.
> Требует сбора реальных бенчмарков по каналам (Show HN, Reddit, PH, README conversion).
> Файл когда будет готово: `agency_business/02_Services/COST_OF_INACTION_MATRIX.md`

**27. Асимметрия риска — не гарантия, а перекладывание риска**
Стандартная гарантия звучит как обещание: "найдём 5 gaps".
Правильная формулировка — асимметричный риск: ты рискуешь нулём, мы рискуем $500.
Если не найдём — возвращаем полностью. Ты физически не можешь потерять на этой сделке.

**28. Статусный артефакт: публичный кейс**
Мы публикуем кейс (с разрешения клиента) — клиент становится его частью.
"I had my GTM audited. Here's what they found" — это пост который резонирует
с тысячами фаундеров в той же ситуации.
Для OSS-фаундера это: личный бренд + публичная рефлексия + сигнал инвесторам
что он системно думает о росте, не только о коде.

**29. Реалистичность для одного человека без команды**
30-дневный план написан под solo-фаундера. 35 минут в день.
Каждое действие описано до уровня "что именно сделать, каким инструментом, сколько времени".
Не "улучши README" — а "замени строки 1-3 на вот этот текст. 20 минут."
Страх "я получу план но не смогу его исполнить" — снят заранее.

**30. Покрытие 5 из 8 этапов Job Map — то чего не делают другие**
Большинство агентств начинают с Execute (исполнение).
Мы начинаем с Define (переформулируем задачу) и Locate (собираем данные)
до того как дать первую рекомендацию. Это значит рекомендации основаны на данных,
а не на интуиции подрядчика который видит продукт первый раз.
