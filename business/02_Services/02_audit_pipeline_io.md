# GTM Audit — Pipeline I/O
> Для кого: инженеры внутри агентства.
> Цель: понять что входит и что выходит на каждом этапе delivery.

---

## Схема пайплайна

```
CLIENT DATA
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│  STAGE 0: BRIEF                                         │
│  IN:  GitHub, PH, blog, socials (публичные данные)      │
│  OUT: Заполненный бриф — известное / неизвестное        │
└─────────────────────┬───────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────┐
│  STAGE 1: POSITIONING                                   │
│  IN:  Бриф + публичные данные                           │
│  OUT: ICP, struggling moment, конкурентная рамка,       │
│       messaging hierarchy, первичные gaps               │
└─────────────────────┬───────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────┐
│  STAGE 2: PACKAGING AUDIT                               │
│  IN:  README, PH listing, все точки входа               │
│  OUT: Постатейный аудит README, список Travel Costs,    │
│       Time to First Value метрика                       │
└─────────────────────┬───────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────┐
│  STAGE 3: LAUNCH AUDIT                                  │
│  IN:  Все каналы где продукт появлялся                  │
│  OUT: Что сработало / что нет / что не пробовали вообще │
└─────────────────────┬───────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────┐
│  STAGE 4: GAP ANALYSIS                                  │
│  IN:  Выходы Stage 1 + 2 + 3                            │
│  OUT: Таблица gaps: фаза / проблема / импакт / действие │
│       Минимум 5. Каждый gap — конкретный и измеримый.   │
└─────────────────────┬───────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────────────────┐
│  STAGE 5: DELIVERABLES                                  │
│  IN:  Gap analysis                                      │
│  OUT: Notion-отчёт + Loom script + 30-Day Plan          │
└─────────────────────────────────────────────────────────┘
```

---

## Детальный I/O по каждому этапу

---

### STAGE 0 — Brief

| | |
|---|---|
| **Вход** | URL продукта + имя фаундера |
| **Процесс** | Сбор данных: GitHub, PH, blog, соцсети, конкуренты |
| **Инструменты** | WebSearch, WebFetch, существующий конкурентный анализ |
| **Выход** | `01_CLIENT_BRIEF.md` — таблица: поле / данные / источник / статус (✅ known / ❓ unknown) |
| **Критерий** | Все 6 полей брифа закрыты — либо данными, либо обоснованной гипотезой |

---

### STAGE 1 — Positioning

| | |
|---|---|
| **Вход** | `01_CLIENT_BRIEF.md` + публичные данные |
| **Процесс** | 4 скила: `problem-definition` → `value-architecture` → `positioning-messaging` → `neuro-foraging-cjm` |
| **Инструменты** | Скилы агентства |
| **Выход** | `02_POSITIONING_ANALYSIS.md` содержащий: struggling moment, ICP-профиль, 3-слойный value audit, positioning canvas, messaging hierarchy, CJM с точками отвала |
| **Критерий** | Найден ≥1 gap на каждом из 3 ценностных слоёв |

---

### STAGE 2 — Packaging Audit

| | |
|---|---|
| **Вход** | README (raw), PH listing, все внешние ссылки продукта |
| **Процесс** | Построчный аудит README против 11 GTM-проверок нашего анализатора + скил `github-traction-audit` |
| **Инструменты** | README Analyzer (наш инструмент), скил `github-traction-audit` |
| **Выход** | `03_PACKAGING_AUDIT.md` содержащий: скор по 11 проверкам, Time to First Value (шаги), список Travel Costs по приоритету, конкретные правки с примерами |
| **Критерий** | Каждый gap — конкретная строка/секция README или отсутствующий элемент. Не "улучшить описание" а "добавить GIF демо после заголовка" |

---

### STAGE 3 — Launch Audit

| | |
|---|---|
| **Вход** | Список каналов из брифа + данные о запусках (PH, blog, socials) |
| **Процесс** | Скилы `launch-strategy` + `ph-launch-readiness` |
| **Инструменты** | Скилы агентства |
| **Выход** | `04_LAUNCH_AUDIT.md` содержащий: таблица каналов (использован / результат / оценка), что не использовано и почему стоит, оценка storytelling-качества |
| **Критерий** | Найдены ≥2 канала с высоким потенциалом которые не использовались |

---

### STAGE 4 — Gap Analysis

| | |
|---|---|
| **Вход** | `02_POSITIONING_ANALYSIS.md` + `03_PACKAGING_AUDIT.md` + `04_LAUNCH_AUDIT.md` |
| **Процесс** | Синтез всех находок, дедупликация, приоритизация по импакту |
| **Инструменты** | — |
| **Выход** | `05_GTM_GAPS.md` — единая таблица: № / Фаза / Gap / Импакт (H/M/L) / Конкретное действие / Ожидаемый эффект |
| **Критерий** | ≥5 gaps. Каждый: измеримый, устранимый фаундером без команды, с конкретным действием |

---

### STAGE 5 — Deliverables

| | |
|---|---|
| **Вход** | `05_GTM_GAPS.md` |
| **Процесс** | Написание отчёта + плана |
| **Инструменты** | — |
| **Выход 1** | `06_REPORT.md` — Notion-отчёт: executive summary (3 строки), топ-5 gaps с объяснением, полная таблица, рекомендации |
| **Выход 2** | `07_30DAY_PLAN.md` — 4 недели: неделя / действие / метрика успеха |
| **Критерий** | Фаундер может начать выполнять план в день получения без дополнительных вопросов |

## Что НЕ входит в этот аудит

- Реализация: мы не делаем сайт, не пишем контент, не ведём соцсети
- Технический аудит продукта
- Финансовый анализ
- Outreach-кампании

Это диагноз. Не лечение.
